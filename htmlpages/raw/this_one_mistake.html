Introduction

Recently, a colleague reached out for help with some code that was taking days to run. They had considered whether the piece of problematic code should simply be dropped from the program. First, we explored Rust as an option, and I shared some of my thoughts here. However, when reviewing the R code, we discovered a few optimizations that made the code more than 1,000x faster. There was one piece of code that was particularly problematic. So, I wanted to write this post to describe the problem as well as my suggestion for fixing it.

The Code

In the code sample below, we first create a data.table with two columns of strings. Most of the strings are just single letters of the English alphabet, but some words are mixed in. After setting up the data, we try to define a variable res. This variable calculates the approximate string distance between the left and right vectors of each row. If one of the words only has one letter, then the first character in both strings are compared for equality. This control logic is implemented in a fcase statement. However, the code throws an error: 

library(data.table)
library(stringdist)

# Sample data
nrec <- 1000
options <- c("CAT", "HAT", "THAT", "SPAT", "SCENTED", "APRON", LETTERS)
dt <- data.table(ID = 1:nrec, left = sample(options, nrec, TRUE), right = sample(options, nrec, TRUE))

# String distance: check for equality if one string is only one character wide
fcase_by <- function(left, right) {
  fcase(
    nchar(left) == 1 | nchar(right) == 1,
    as.numeric(substr(left, 1, 1) != substr(right, 1, 1)),
    default = stringdist(left, right)
  )
}

dt[, res := fcase_by(left, right)]

The Problem

The problem is with the way the default argument of fcase is being used. It is designed for hard-coded patterns, like 0 or 1. Instead, it is being used as control flow for multiple functions. The code I reviewed did not look like this, but I suspect it did at one point. Why? Because the actual code finds a workaround: 

dt[, res := fcase_by(left, right), by = ID]

While this fixed the error, it ruined the performance. The by operator is typically reserved for groupwise operations. Here, it is used on a column that is unique for each row of data. Thus, each row is evaluated indepdendently, which forces the entire function to be non-vectorised. Most R users have been told to avoid for loops because they are slow. Let's see how well common wisdom holds up: 

# Same logic, but in for loop
sdist_loop <- function(left, right) {
  n <- length(left)
  out <- rep(NA, length(left))
  for (i in n) {
    if (nchar(left[i]) == 1 | nchar(right[i]) == 1) {
      out[i] <- as.numeric(substr(right[i], 1, 1) != substr(left[i], 1, 1))
    } else {
      out[i] <- stringdist(left, right)
    }
  }
  out
}

bench_by_nrec <- function(nrec) {
  options <- c("CAT", "HAT", "THAT", "SPAT", "SCENTED", "APRON", LETTERS)
  dt <- data.table(ID = 1:nrec, left = sample(options, nrec, TRUE), right = sample(options, nrec, TRUE))

  bench <- data.table(microbenchmark(
                          "By + fcase" = dt[, res := fcase_by(left, right), by = ID],
                          "For loop" = dt[, res2 := sdist_loop(left, right)],
                          times = 10
  ))
  bench[, nrec := nrec]
  bench[, time := time / 1e9]
  bench[, list(expr, nrec, time)]
}

out <- do.call(rbind, lapply(seq(10000, 50000, by = 10000), bench_by_nrec))

p <- ggplot(out, aes(x = nrec, y = time, group = nrec)) +
  geom_line() +
  geom_smooth(method = "lm") +
  theme_minimal() + 
  ylab("Time (s)") +
  xlab("Number of rows")

svglite::svglite("dt_strdist_bench.svg", width = 6.0, height = 4)
plot(p)
dev.off()

Ouch. The for loop starts off 200x more performant, and by 50,000 records the difference approaches 1,000x. As it turns out, the for loop is the most performant option I have tried to date. I had originally assumed a vectorized ifelse would have been the fastest. I still have some digging to do to understand why the for loop did so well. My current thinking is that multithreading is attempted in both data.table and stringdist, which becomes counterproductive. But, that is a topic for another day. For now, let's focus on why the original solution was so slow.

Most R functions are little more than bindings for functions in low-level languages, typically C and C++. One of the reasons these languages are faster is because they manage memory efficiently. To automate memory management for R users, R objects include some extra overhead. Some of this overhead is the same for all R objects, no matter their size. Furthermore, R activates additional memory optimizations once an R object grows large enough. The below code example demonstrates this: 

objs <- list("single int" = 1, "two ints" = 1:2, "first 10,000 ints" = 1:10000)
lapply(objs, function(x) c("Total size" = object.size(x), "Size per element" = object.size(x) / length(x)))
$`single int`
      Total size Size per element 
              56               56 

$`two ints`
      Total size Size per element 
              56               28 

$`first 10,000 ints`
      Total size Size per element 
      40048.0000           4.0048

A list of one element requires just as much memory as a list of two element (56 bytes). Even better, a list of 10,000 elements is 7 times more memory efficient than a list of two (4 bytes per element). In the rowwise by function, we make one object for each row, then combine them into a single list. This follows logically from the functionality of the by operator. If you are not convinced, modify the function to print something each time it is called and count how many lines are printed out. In the for loop, we create one object per thread and then fill in the values. Data.table uses multithreading in certain conditions, so the function would be called once on each worker. You might think the memory efficiency does not matter if the end object is the same size. But memory allocation, memory deallocation, and combining objects each incur their own costs. There is also the overhead of multithreading itself: multithreading is only faster if the cost of sending data from the main thread to a worker thread and back again is relatively small. In this case, memory is passed back and forth very inefficiently, so we would have been better off without multithreading.

Conclusion

Languages like R and Python have fluorished precisely because trading performance for ease of use is often worth it. However, when performance matters, the hidden optimizations become more annoying and less helpful. My advice? Assume nothing and benchmark everything. The microbenchmark package in R is great for testing small units of code. Hadley's Advance R book has a section on performance that is also worth browsing. In this case, we found enough performance that we might not need a low level language after all. It is also important to understand the tools you are using. I have never regretted reading documentation for anything I use a lot: code editors, R packages, anything. Finally, definitely consider adding a low-level language to your toolkit. Not only can you integrate C, C++, and Rust code into R and Python, but they will teach you concepts about coding you would not otherwise learn. I probably would have made a similar mistake had I not began learning Rust.
